# WCAG 2.1 AA Audit — Design Specification

**Issue:** [#311](https://github.com/devify-me/devify-ui/issues/311)  
**Date:** 2026-04-05  
**Status:** Design approved

---

## Goal

All `dvfy-*` components meet **WCAG 2.1 Level AA** compliance. Verify via automated tests (axe-core) + manual a11y audit. Production-grade component library requires this before Build-Status ships to real users.

## Problem

- No accessibility audit performed on existing 50+ components
- Keyboard navigation, ARIA roles/labels, focus management not systematically verified
- No automated test coverage to prevent regressions
- Color contrast audit exists but not integrated into component testing

## Scope

### In Scope
- Audit all existing `dvfy-*` components for WCAG 2.1 AA compliance
- Add axe-core integration to test suite
- Test keyboard navigation, focus management, ARIA roles/labels
- Fix every WCAG failure found
- Add CI job to prevent regressions

### Out of Scope
- WCAG AAA (higher standard, not required)
- Internationalization or RTL support (separate effort)
- Third-party component dependencies

## Architecture

### Three-Layer Testing Strategy

#### Layer 1: Axe Utility Helper
Create reusable test utility (`utils/axe-test.js`) that wraps axe-core for consistency:

```javascript
export async function checkA11y(element, context = {}) {
  const results = await axe(element, context);
  expect(results).toHaveNoViolations();
}
```

- Wraps `chai-a11y-axe` (already a dependency via @open-wc/testing)
- Single-line assertions in any test
- Clear error output on violations

#### Layer 2: Functional Tests + Axe Checks
Augment existing component tests (~20 critical Tier 1-2 components) with axe assertions:

**Pattern:**
```javascript
describe('dvfy-button', () => {
  it('renders with variant', async () => {
    const el = await fixture(html`<dvfy-button variant="danger">Delete</dvfy-button>`);
    expect(el.getAttribute('variant')).to.equal('danger');
    await checkA11y(el);  // ← Add this to each test
  });
});
```

**Scope:** Button, Input, Checkbox, Radio, Switch, Textarea, Select, Dropdown, Tabs, Modal, Drawer, Breadcrumb, Pagination, Accordion, Table, Nav, Sidebar, Theme-Switcher, Alert, Toast.

#### Layer 3: Deep A11y Test Suite
Separate test files (`*.a11y.test.js`) for complex patterns that need manual verification. Axe alone cannot catch:
- Focus trap (modal stays focused inside)
- Keyboard navigation sequences (arrow keys in dropdowns)
- ARIA state transitions (aria-expanded when toggle opens)
- Announce-on-change patterns (table row selection announces)

**Scope:** Modal, Dropdown/Select, Table, Nav (skip links, active announcements), Drawer (focus containment).

### Component Classification

#### Tier A: Interactive + Axe checks (add to existing tests)
Components where interaction + a11y are critical. Add axe assertion to each test scenario.

- **Form controls:** button, input, checkbox, radio, switch, textarea, select, dropdown
- **Complex patterns:** modal, drawer, tabs, accordion, table, nav, breadcrumb, pagination
- **State-bearing:** theme-switcher, alert, toast

#### Tier B: Dedicated a11y tests (new files)
Complex interaction patterns where keyboard/focus behavior matters most. Requires manual test scenarios.

- **modal.a11y.test.js** — focus trap, Escape key close, aria-modal, backdrop interaction
- **dropdown.a11y.test.js** — arrow key navigation, auto-focus on open, Enter/Space activation
- **select.a11y.test.js** — same as dropdown (may be same file if both use similar pattern)
- **table.a11y.test.js** — keyboard row/cell navigation, row select announcements
- **nav.a11y.test.js** — skip links, active link announcements, semantic structure
- **drawer.a11y.test.js** — focus containment, swipe/Escape close, aria-modal

#### Tier C: Visual/Manual audit only (no test code changes)
Presentational components where WCAG compliance is primarily visual (color contrast, text alternatives). No interactive state to test.

- badge, tag, card, empty, loader, progress, tooltip, avatar, hamburger, breadcrumb, pagination, scroll-progress, marquee-scroll, etc.
- **Audit approach:** Walk through catalog, verify token usage, check for hardcoded colors, ensure text alternatives for icons.

## Testing Patterns

### Axe Check Pattern (Layer 2 & 3)

```javascript
import { fixture, html, expect } from '@open-wc/testing';
import { checkA11y } from '../utils/axe-test.js';
import './dvfy-button.js';

describe('dvfy-button', () => {
  it('renders with default attributes', async () => {
    const el = await fixture(html`<dvfy-button>Click me</dvfy-button>`);
    expect(el.textContent.trim()).to.equal('Click me');
    await checkA11y(el);  // ← One line per test
  });

  it('renders disabled state', async () => {
    const el = await fixture(html`<dvfy-button disabled>Disabled</dvfy-button>`);
    expect(el.getAttribute('aria-disabled')).to.equal('true');
    await checkA11y(el);
  });
});
```

### A11y-Specific Test Pattern (Layer 3)

```javascript
describe('dvfy-modal accessibility', () => {
  it('traps focus inside modal when open', async () => {
    const el = await fixture(html`
      <dvfy-modal open>
        <button>Close</button>
      </dvfy-modal>
    `);
    const closeBtn = el.querySelector('button');
    closeBtn.focus();
    
    // Verify Tab wraps to first focusable inside
    const focusableElements = el.querySelectorAll('[tabindex], button, input');
    expect(document.activeElement).to.equal(closeBtn);
    
    // Simulate Tab key
    closeBtn.dispatchEvent(new KeyboardEvent('keydown', { key: 'Tab' }));
    // Focus should wrap to first element
    expect(focusableElements.length).toBeGreaterThan(1);
  });

  it('closes on Escape key', async () => {
    const el = await fixture(html`<dvfy-modal open>Content</dvfy-modal>`);
    const initialState = el.getAttribute('open');
    
    el.dispatchEvent(new KeyboardEvent('keydown', { key: 'Escape' }));
    await el.updateComplete; // Web Component update
    
    expect(el.getAttribute('open')).to.not.equal(initialState);
  });

  it('has aria-modal role', async () => {
    const el = await fixture(html`<dvfy-modal open>Content</dvfy-modal>`);
    expect(el.getAttribute('role')).to.equal('dialog');
    expect(el.getAttribute('aria-modal')).to.equal('true');
  });
});
```

## Key Design Decisions

| Decision | Rationale |
|---|---|
| Axe checks in existing tests | Catches regressions on interactive components; low cost (one line) |
| Separate a11y files for complex patterns | Manual sequences (keyboard nav, focus) need scenario testing that axe can't provide |
| No tests for presentational components | Badge, card, tag are pure CSS; axe finds color contrast issues in separate audit |
| Reusable axe helper | Consistency, easier to modify behavior later (e.g., add custom rules) |
| No new npm scripts | `npm test` already runs all `*.test.js` files; no special configuration needed |

---

## Risks & Mitigation

| Risk | Mitigation |
|---|---|
| Test suite becomes slow with axe checks | Axe runs in browser (parallel); typical overhead is 2-3 seconds for full suite |
| A11y tests are brittle (depend on timing) | Use `@open-wc/testing` utilities (fixture, oneEvent) for deterministic DOM updates |
| Coverage gaps (some components untested) | Tier C components still audited manually; focus on high-impact interactive components first |
| Axe false positives | False positives documented in test comments; suppress with `{ rules: { 'rule-id': { enabled: false } } }` if justified |
